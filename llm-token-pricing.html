01 · 基础概念

Token 是什么？

大模型不直接「读」文字，而是把文本切成一个个 token——可以理解成语言的最小计价单位。

📝 中文 vs 英文分词

中文通常 1-2 个字一个 token，英文则是一个常见词或部分词一个 token：

🇨🇳 中文
"今天天气真好"
今天 → 天气 → 真 → 好
6 个字符 → 4 个 token
🇬🇧 英文
"The weather is great"
The → weather → is → great
19 个字符 → 4 个 token

💰 Token = 定价原子

API 按 token 收费——你输入的文本（prefill）和模型生成的文本（decode）都算。但它们的计算成本完全不同，这就是为什么输入和输出价格差好几倍。

02 · 核心机制

推理的两幕戏

每次 LLM 调用其实分为两个阶段——prefill 和 decode。理解这两者，就理解了定价的根本逻辑。

⚡

第一幕：Prefill（预填充）

也叫 "prompt processing" 或 "encoding"
输入： "今天 天气 怎么 样？ 请 写 一首 诗"
↓ 所有 token 同时进入，并行处理 ↓
🧮 矩阵乘法 · 并行计算 · GPU 大展拳脚
✅ 优势： 大量 token 一次性并行处理，GPU 利用率高
⚠️ 瓶颈： 显存带宽——要把整个输入一次性搬进 GPU
🐢

第二幕：Decode（解码生成）

也叫 "auto-regressive generation"
输出：
今天 ← 第1步
天气 ← 第2步（需要等上一步结束）
晴 ← 第3步
朗 ← 第4步（一个接一个 🔗）
...
⛓️ 串行生成 · 每步只产一个 token · 必须等上一步
✅ 优势： 灵活——生成内容长度可变，质量高
⚠️ 瓶颈： 每步只算一个 token，GPU 大部分时间在"等显存"

一图胜千言

🚀
Prefill
输入全部 token
一次性并行处理
计算密集
VS
🐢
Decode
逐个生成 token
必须串行等待
内存密集
03 · 算账时间

为什么价格差这么多？

一句话：输出比输入贵，是因为 decode 的计算效率远低于 prefill——GPU 大部分时间在"等人"而不是"干活"。

🏭 工厂流水线类比

Prefill = 批量生产线
100 个零件一起上流水线
机器全速运转 ⚡
→ 每个零件摊到的成本极低
Decode = 手工作坊
做完一个才能做下一个 🔗
机器等师傅、师傅等机器
→ 每个零件摊到的成本很高

📐 数学本质：FLOPs 视角

每个 token 经过 Transformer 的计算量大致是：

每个 token 的 FLOPs ≈
2 × 参数量 × 上下文长度
（忽略归一化和激活函数的微小贡献）

这里的「上下文长度」是关键。prefill 阶段，上下文包含你输入的 所有 token；decode 阶段，每生成一个新 token，上下文就 +1，计算量随输出长度增长。

💡 直觉理解： 生成第 1000 个 token 时，模型要"回顾"前面 999 个 + 所有输入 token，计算量是在不断膨胀的。

05 · 实战优化

开发者的省钱指南

5 条立竿见影的策略，帮你在不牺牲质量的前提下大幅降低 API 开销：

1

✂️ 精简 Prompt

每减少 100 个输入 token，就省下相应的 prefill 费用。去掉多余的客套话、精简示例、用更紧凑的指令风格。对于高频调用的 API，这一点日积月累下来非常可观。

2

💾 利用 Prompt Caching

把 system prompt 和固定的上下文放前面，变化的部分放后面。缓存命中后输入价格可降 50%。Claude、GPT-4o 均支持此特性。

3

🎯 限制输出长度

设置合理的 max_tokens 上限。不要让它自由发挥——每多生成 1000 个 token，decode 费用比 prefill 贵 3-5 倍。

4

🔀 按场景选模型

不是所有任务都需要最强模型。简单的分类、提取、格式化任务用便宜模型（如 Claude Haiku / GPT-4o-mini），只有复杂推理才出动旗舰。一个简单的路由规则就能省下大量开销。

5

📦 批处理 + 异步调用

多个独立请求可以并行发出，无需等待前一个完成。部分 API 还支持 Batch API，用延迟换折扣（通常 50% off），适合非实时场景。
